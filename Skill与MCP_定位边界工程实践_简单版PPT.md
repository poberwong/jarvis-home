# Skill 与 MCP：定位、边界与工程实践
副标题：从 Anthropic 官方文档到一线落地经验  
汇报人：美团酒店旅行技术部（爸爸）

---

## 第1页｜开场：我们到底在讨论什么？
- 背景：Agent 工程进入“可落地”阶段，能力分层成为关键
- 核心问题：Skill 和 MCP 到底分别解决什么问题？
- 目标：澄清定位、划清边界、沉淀可执行实践

---

## 第2页｜一句话定义（先给结论）
- **MCP**：原子能力提供层（Tools / 资源 / 标准接口）
- **Skill**：任务工作流与方法层（步骤、策略、最佳实践）
- **知识库**：事实与数据层（领域信息、上下文材料）

> 记忆口诀：
> MCP = 能力积木；Skill = 组装说明书；知识库 = 原材料清单

---

## 第3页｜官方框架对照：厨房比喻
- Anthropic 官方常用类比：
  - MCP 像“专业厨房”：灶台、刀具、原料接口（可调用能力）
  - Skill 像“菜谱”：按场景组织步骤与决策（如何做）
- 价值：把“会做事”拆成“能做什么 + 怎么组织”

---

## 第4页｜Skill vs 知识库：程序性知识 vs 陈述性知识
- **Skill（Procedural）**：知道“怎么做”
  - 决策流程
  - 失败重试策略
  - 工具编排顺序
- **知识库（Declarative）**：知道“是什么”
  - 业务规则
  - 产品资料
  - 历史事实

结论：两者是互补关系，不是替代关系。

---

## 第5页｜常见反模式：把基础能力硬塞进 Skill
- 现象：Skill 里混入大量 fetch/CLI/环境安装碎片
- 问题：
  1. 职责混乱：知识层侵入能力实现层
  2. 难复用：同一能力被多个 Skill 重复拷贝
  3. 难维护：工具升级要改 N 份脚本
  4. 不稳定：执行链路长且脆弱

---

## 第6页｜为什么这会影响执行效果？
- 工具调用语义不统一，Agent 难以稳定选择
- 错误处理分散在脚本碎片中，失败恢复不可预测
- 上下文噪声增加，模型决策成本上升
- 结果：看起来“能跑通”，但线上波动大、迭代慢

---

## 第7页｜推荐分层架构（可直接落地）
- **L1 能力层（MCP）**：
  - 封装原子工具：search/fetch/browser/file/db/api
  - 提供统一输入输出与错误码
- **L2 编排层（Skill）**：
  - 任务 SOP
  - 决策条件、回退路径
  - Prompt 模板与质量检查
- **L3 知识层（KB）**：
  - 业务事实、策略文档、案例样本

---

## 第8页｜工程实践清单（团队可执行）
1. **先下沉能力再写 Skill**：基础操作优先 MCP 化
2. **Skill 不写重逻辑**：保持“指令化、轻脚本”
3. **统一 Tool Schema**：输入输出、异常码标准化
4. **观测先行**：记录 Tool 成功率、重试率、耗时
5. **灰度迭代**：先单场景验证，再扩展到多任务

---

## 第9页｜边界不是教条：目标导向的辩证实践
- 第一性原则：架构是手段，业务目标是目的
- 可以灵活，但要有约束：
  - 临时混层可以接受（应急）
  - 中长期必须回归分层（治理）
- 判断标准：
  - 是否提高稳定性？
  - 是否提升复用率？
  - 是否降低维护成本？
